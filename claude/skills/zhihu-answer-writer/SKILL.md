# 知乎高分回答写手

帮用户把一个知乎问题，写成一篇有"高赞气质"的回答——开头抓人、结构清晰、有观点有案例、排版舒服、配图到位、读起来像真人答主而不是 AI。

知乎的残酷现实：**信息流只露前两三行，排序由互动（赞/藏/评/完读率）驱动，读者只认"具体的人 + 具体的信息"**。这三条决定了一切打法。完整的套路库在 [references/zhihu-high-score-patterns.md](references/zhihu-high-score-patterns.md)——动笔写每个阶段前都该回去对照，它解释了每条技巧"为什么有效"，比死记规则重要。

## 为什么要多轮确认

知乎回答是高度个人化的创作：用什么立场、走什么结构、用谁的经历当案例，都得贴合用户本人。一口气写完整篇再让用户推翻，既浪费又难改。所以拆成四个卡点，**每步定下来再往下走**——越早对齐方向，返工越少。不要图快跳过确认，那样产出的多半是用户用不上的"正确的废话"。

## 工作流程

### 第 0 步：拿全问题 + 联网调研话题（动笔前必做）

先确认拿到的是**完整问题**。知乎问题标题下面常有一段"问题补充/描述"，提问者会在那里交代背景和真实诉求，它经常彻底改变正确答法（标题问"该不该考研"，补充里写"我已工作两年想转行"——答法完全不同）。如果用户只给了标题，问一句"这题下面有没有问题补充？把它一起发我"，没有就按标题答。

拿全问题后，别急着写。用 WebSearch / WebFetch 把这个话题摸清楚，这是"具体信息"的来源，也是和纯 AI 空泛回答拉开差距的关键：

- 这个话题的**事实、数据、时间线**——有具体数字的回答才可信。
- 已有的**高赞回答怎么答的**——不是抄，是看清楚这题的读者想要什么、已有答案的空白在哪，找差异化角度。
- 可用的**真实案例 / 争议点 / 反常识结论**。

调研要快而准（2-4 次检索即可），抓到能支撑回答的硬料就停。把关键发现记在心里，后面每一步都用得上。

> 如果用户明确说"别联网、就用你知道的写"，跳过这步，但要提醒：缺少具体数据的回答在知乎上偏弱。

### 确认 ① 问题拆解 + 立场 + 钩子

调研完，先和用户对齐**这题怎么答**。一次性给出下面三样，让用户挑/改：

1. **问题归类与拆解**：这是六类里的哪类（知识科普 / 个人经验 / 情感共鸣 / 测评推荐 / 争议辩论 / 职业行业，见套路库第 9 节），读者真正想要的是什么，这题的"题眼"在哪。
2. **答主立场 / 人设**：建议用哪种人设（第一人称亲历 / 专家科普 / 犀利观点），以及具体怎么立——用用户的什么身份、什么经历。**这步要问用户**：他本人和这话题是什么关系？有没有真实经历可用？人设必须真实，编造的经历会崩。
3. **2-3 个候选开头钩子**：写出来给用户选，每个标明用的是哪种钩子（反常识/亮证/数据/悬念/场景/总述）。

用 `AskUserQuestion` 让用户在候选立场和候选钩子里选，或给开放反馈。**定下来再进下一步。**

### 确认 ② 结构 / 大纲

立场和钩子定了，搭骨架。从套路库第 4 节选一个适配的结构模板（结论先行 / 故事+道理 / 清单体 / 层层递进 / 对比体 / 总分总），列出**逐段大纲**：

- 每一段/每一节讲什么、放什么案例或数据、承担什么作用。
- 标出"观点"主线和支撑它的"案例/数据"血肉——确认两者都有，别变成纯干货或纯故事。
- 标出预计的情绪曲线高点和可能的金句位置。
- **定详略**：长度随问题类型走，不是越长越高赞。情感共鸣题贵在精炼戳心，硬撑长度反而稀释情绪；干货/测评/科普题才需要长篇和小标题分块。先和用户对一下大概篇幅。

把大纲给用户过目，调整后再往下。**不要在大纲没定时就开始写正文。**

### 确认 ③ 正文初稿

按大纲写完整初稿。写的时候：

- **严格对照套路库**：开头第一句给钩子不铺垫；短段短句、加粗只给重点、该分点分点；金句放在情绪高点；结尾先收主旨再做利他式互动引导。
- **此时先不插真图**，但在该配图的位置用 `【配图位：画什么】` 占位标出来，方便下一步统一规划。
- **写完立刻做去 AI 味自检**（套路库第 11 节）：删掉"综上所述/此外/值得注意的是/研究表明"等 AI 标志词，换成口语；加入具体细节、个人化措辞、适度不完美和明确立场。交给用户的初稿就该是去过 AI 味的，别让用户替你擦。

把初稿给用户读，收集修改意见，改到用户认可。

### 确认 ④ 配图方案

正文定稿后，规划配图。按 [references/illustration-playbook.md](references/illustration-playbook.md) 做：

- 针对每个 `【配图位】`，做一张表定四件事：放哪、画什么、什么类型、用哪条路子出图（**A: gpt-image 真生成** 还是 **B: 推荐真实网图**）。
- **示意图/信息图/封面/概念图 → 路子 A**：调用 `gpt-image` skill 真生成。
- **真实统计数据/实物/截图 → 路子 B**：联网找现成图，给链接和来源，标版权提示。绝不用 gpt-image 伪造"真实数据图"。
- **先把这张配图方案表给用户拍板**，再去生成/找图——图返工贵。

确认后执行出图，把生成的图片文件或网图链接配齐。

### 第 5 步：交付成品

把正文 + 配图组装成最终 markdown：

- 图片在正文里按 illustration-playbook 的格式落位（生成图给文件路径，网图给链接+图注+版权提示）。
- 末尾附一段**简短交付说明**：这篇用了什么人设/结构/钩子，哪些图是 gpt-image 生成的、哪些建议用户自己换真实图、为什么。让用户知道怎么用、怎么微调。
- 提醒用户：发布前通读一遍，把人设细节（具体经历、数字）替换成自己真实的，这样才站得住。

## 几条贯穿始终的硬约束

这些是所有来源最一致的共识，违反基本与"高赞"无缘：

1. **开头第一句必须有信息或钩子，禁止铺垫。** 信息流只露前两三行。
2. **"观点 + 案例"双驱动。** 纯干货空泛，纯故事没主见。
3. **短段、加粗重点、分点**是基本功，不做就别想高赞。
4. **主动去 AI 味**：删 AI 标志词，加具体细节、个人立场、口语节奏。这是 AI 起草的内容能否过关的分水岭。
5. **结尾互动引导要利他**，别用"码字不易求三连"这类模板话术。
6. **人设必须真实**：用用户自己的身份和经历，不替他编造可被戳穿的经历。

## 灵活度

这是一个**创作类**流程，不是死板的检查表。用户如果说"别啰嗦，直接给我写一版"，可以压缩确认步骤、一次性给初稿再迭代；如果用户是老手只想要钩子和结构建议，就别硬塞全流程。四个卡点是默认路径，不是枷锁——核心是套路库里的原理和"具体的人+具体的信息"这条魂，那些不能丢。
