# technical-zh：简洁思辨型（技术博客 / 科普 / 深度文章）

目标读者：技术人、写技术的人、想看懂技术的非技术人。
参考调性：阮一峰、刘未鹏、王垠（语气克制版）、Paul Graham 中文化。

## 核心立场

- 准确 > 漂亮
- 例子 > 抽象
- 推理 > 断言
- **不打官腔，不用学术腔**

## 句法偏好

- **中短句为主**（10-18 字），偶尔一句长句铺陈逻辑链
- 段落短（3-5 句一段）
- 段首多样：观察句 / 问句 / 反例句 / 时间状语开头都可

## 必须出现

- 至少 1 个具体例子（代码片段 / 真实工具名 / 案例公司名 / 数字 / API 名）
- 至少 1 处作者立场（"我倾向 X 而不是 Y"、"这个做法有个问题"）
- 至少 1 个**反直觉**点（颠覆常识 or 颠覆作者自己上文的判断）

## 禁用清单

| 禁用 | 因为 |
|------|------|
| 综上所述 / 总而言之 / 综上 | 学术八股 |
| 不仅 X 而且 Y | 完美对仗，太工整 |
| 值得注意的是 / 不可否认 | 假权威引导词 |
| 实际上 / 其实 + 限定词堆叠 | 稀释信息 |
| 进行 / 做出 / 加以 + 名词 | 弱化动词 |
| em-dash 单段 > 1 | AI 标志 |
| 三联词 | 工整就是 AI |

## 鼓励的手法

- **从一个具体问题出发**（不要"X 在当今社会的重要性"开场）
  - ✅ "上周我在 grep 一个 50G 的 log 文件，ripgrep 比 grep 快 12 倍。这件事让我想了好几天 ..."
  - ❌ "在当今的大数据时代，文本搜索工具的性能至关重要"

- **承认不确定 / 反例**
  - ✅ "这个做法在 X 场景下成立，但 Y 场景下我没想清楚"
  - ✅ "也可能我说错了"

- **用对比代替形容**
  - ✅ "它比 Postman 快 5x，比 curl 慢一点"
  - ❌ "它非常高效"

## 节奏样本（写一段技术博客像这样）

> 类型推断这件事容易被神化。它的本质是反向求解，不是魔法。
>
> TypeScript 的类型推断也就那回事。你写 `const x = 1`，编译器知道 `x` 是 `1`（不是 `number`，是字面量 `1`）。给 `let`，才是 `number`。这个差异困住过我很多次。
>
> 上周我帮一个 PR 改类型，发现作者把所有 `const` 都改成 `let`。他想要更"灵活"。结果整个文件的类型推断全废了。
>
> 推断有边界。它不是魔法。它只是反向求解。理解这一点之后，你才能写出好用的类型。

注意这段：

- 起手就是观察句，没有元评论
- 4 段，最短 9 字，最长 38 字——burstiness 高
- 有具体例子（`const x = 1` 这种）
- 有立场（"被神化"、"也就那回事"）
- 有 anecdote（"上周帮 PR 改类型"）
- 收尾不"展望未来"，给一个克制的观察句
